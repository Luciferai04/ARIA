01 Abstract
ARIA (Agentic Research Intelligence Assistant) is a fully autonomous, multi-node LangGraph agent designed to
transform how researchers and founders consume technical literature. A user states a research question in plain
language; ARIA autonomously plans a multi-step investigation, decomposes it into sub-queries, retrieves grounded
context from a ChromaDB knowledge base, executes live web and arXiv searches in parallel, synthesises a
structured report, evaluates its own faithfulness via a self-reflection gate, and iterates if quality falls below threshold
— all within a stateful, multi-turn conversation session.
The project satisfies all six mandatory capstone capabilities prescribed by the course (LangGraph Workflow, RAG
Knowledge Base, Conversation Memory, Self-Reflection, Tool Use, and Deployment) and extends them through a

02 Problem Statement & Motivation
The volume of AI/ML literature has grown at an exponential rate since 2020. Researchers, students, and startup
founders face three compounding problems:
Information Overload
arXiv alone publishes 500+ ML papers per day. Manual triage is impossible at this velocity. Founders and
researchers spend 40–60% of their research time on literature discovery rather than synthesis.
Grounding Gap
Existing LLM chatbots hallucinate citations and fabricate paper titles. Without a retrieval-grounded architecture,
AI-assisted research cannot be trusted for academic or investment decision-making.
Context Fragmentation
Each research query starts from scratch. There is no system that maintains session context, remembers prior
findings, and allows iterative drill-down — the way a human research assistant would.
Why this matters for the author: Soumyajit Ghosh is simultaneously managing two research papers under
review, an ISRO quantum-ML collaboration (AGEIS), and a deep-tech startup (SynthDoc). ARIA is not a
hypothetical tool — it directly accelerates real work and demonstrates the author's capacity to build
production-grade agentic systems.

04 System Architecture Overview
ARIA implements a Plan-Execute-Reflect-Synthesise (PERS) agentic pattern on top of LangGraph's StateGraph
API. Unlike simple RAG chains, the architecture has a planning phase that decomposes queries before any
retrieval occurs, enabling parallel sub-query execution and genuinely multi-source synthesis.
USER INPUT
Plain-language research question via Streamlit UI
M
memory_node
Append turn to conversation history, apply sliding window (k=10)
M

Decompose question → 3–5 sub-queries (LLM + structured output)
M  M  M  (parallel branches)

ChromaDB semantic search — top-k=5 per sub-query

arXiv API + DuckDuckGo web search — live results
M  merge results
answer_node
LLM synthesises grounded answer from all retrieved context
M

M FAIL (score<0.7) → reflect_node → answer_node (max 2 retries)
M PASS

M
STREAMLIT UI
Render structured report: Summary | Findings | Sources | Follow-ups
Key Architectural Decision: Plan-Execute Pattern
Unlike simple router → retrieve → answer chains, ARIA inserts a planner_node that uses structured LLM output
to decompose queries before touching any retrieval. This enables parallel sub-query execution, dramatically
improves coverage on complex questions, and is the single biggest differentiator vs. other student projects in this
cohort.

06 Node Specifications
memory_node
• Reads: question, messages, thread_id
• Writes: messages (append), context_window (sliding window)
• Logic: Append current question as HumanMessage. Trim messages to last k=10 pairs.
• Key rule: MUST run first on every turn — no node bypasses memory.
• Impl note: Use LangChain MemorySaver with thread_id as config key.
planner_node
• Reads: question, context_window
• Writes: sub_queries (List[str]), route (str)
• Logic: Prompt LLM with structured output parser to produce 3–5 sub-queries.
• Route decision: 'retrieve' if KB-answerable, 'tool' if needs live data, 'both' otherwise.
• Structured output: Use Pydantic PlanOutput(sub_queries: List[str], route: str).
• This node is the PERS differentiator — do not skip or merge it.
retrieve_node
• Reads: sub_queries, route
• Writes: retrieved (str), sources (List[str])
• Logic: For each sub-query, query ChromaDB collection (top_k=5). Deduplicate. Concat.
• Conditional: Only runs if route in ['retrieve', 'both'].
• Embed model: sentence-transformers/all-MiniLM-L6-v2 (free, fast, local).
tool_node
• Reads: sub_queries, route
• Writes: tool_result (str)
• Logic: Run arXiv API tool + DuckDuckGo search on each sub-query. Merge results.
• Conditional: Only runs if route in ['tool', 'both'].
• Tools: ArxivQueryRun (langchain-community) + DuckDuckGoSearchRun.
• Output format: Structured string with paper titles, authors, abstracts, URLs.


---PAGE BREAK---

answer_node
• Reads: question, retrieved, tool_result, context_window, reflection_note
• Writes: answer (str), report (dict)
• Logic: Prompt LLM to synthesise from retrieved + tool_result only (no hallucination).
• Output: Structured dict: {summary, key_findings: List, sources: List, follow_ups: List}
• Grounding instruction: 'Only use information explicitly present in the provided context.'
eval_node
• Reads: answer, retrieved, tool_result, eval_retries
• Writes: faithfulness (float), eval_retries (int)
• Logic: Score = overlap of answer tokens with (retrieved + tool_result) context.
• Simple impl: BM25 overlap OR LLM-as-judge with prompt 'Score faithfulness 0.0-1.0'.
• Routing: faithfulness >= 0.7 → save_node; else eval_retries < 2 → reflect_node.
reflect_node
• Reads: answer, question, faithfulness, retrieved
• Writes: reflection_note (str), answer (updated critique)
• Logic: LLM critique prompt — identify unsupported claims, suggest grounded revision.
• Output feeds back into answer_node for second-pass generation.
• Loop guard: eval_retries incremented in eval_node. Max retries = 2.
save_node
• Reads: answer, report, sources, messages
• Writes: messages (append AIMessage with report), context_window (update)
• Logic: Persist final report as AIMessage. Update sliding window.
• Also writes report to session state for Streamlit rendering.

# aria/tools/arxiv_tool.py
from langchain_community.tools import ArxivQueryRun
from langchain_community.utilities import ArxivAPIWrapper
from langchain_core.tools import tool
@tool
def arxiv_search(query: str) -> str:
    """Search arXiv for recent papers. Returns titles, authors, abstracts."""
    wrapper = ArxivAPIWrapper(top_k_results=5, doc_content_chars_max=800)
    return wrapper.run(query)
@tool
def citation_formatter(sources: list[str]) -> str:
    """Format a list of source strings into numbered citation list."""
    return '\n'.join(f'[{i+1}] {s}' for i, s in enumerate(sources))

09 Self-Reflection & Faithfulness Gate
The self-reflection loop is ARIA's quality assurance mechanism. It ensures every answer is grounded in retrieved
context before delivery to the user — directly satisfying Mandatory Capability #4 from the capstone rubric.
# aria/nodes/eval_node.py
EVAL_PROMPT = '''
You are a faithfulness evaluator. Given the ANSWER and the CONTEXT,
score how well the answer is grounded in the context on a scale of 0.0 to 1.0.
Rules:
- 1.0 = every claim in the answer is directly supported by the context
- 0.7 = most claims are supported; minor additions acceptable
- 0.5 = significant unsupported claims present
- 0.0 = answer is entirely unrelated to context (hallucination)
Respond ONLY with a JSON: {"score": , "issues": []}
'''
def eval_node(state: ARIAState) -> ARIAState:
    context = state['retrieved'] + '\n' + state['tool_result']
    response = llm.invoke(EVAL_PROMPT.format(
        answer=state['answer'], context=context[:3000]))
    result = json.loads(response.content)
    state['faithfulness'] = result['score']
    state['eval_retries'] = state.get('eval_retries', 0)
    return state
def route_after_eval(state: ARIAState) -> str:
    if state['faithfulness'] >= 0.7:
        return 'save_node'
    if state['eval_retries'] >= 2:
        return 'save_node'   # force-pass with low-quality flag
    return 'reflect_node'

10 Conversation Memory Design
# aria/nodes/memory_node.py
from langgraph.checkpoint.memory import MemorySaver
from langchain_core.messages import HumanMessage, AIMessage
WINDOW_SIZE = 10   # last 10 message pairs
def memory_node(state: ARIAState) -> ARIAState:
    # Append current turn
    state['messages'].append({'role': 'user', 'content': state['question']})
    # Sliding window — trim to last WINDOW_SIZE * 2 entries
    all_msgs = state['messages']
    state['context_window'] = all_msgs[-(WINDOW_SIZE * 2):]
    return state
# Graph invocation — thread_id isolates sessions
config = {'configurable': {'thread_id': session_id}}
result = graph.invoke({'question': user_input}, config=config)
• MemorySaver is the checkpointer — it persists StateGraph state keyed by thread_id.
• Each Streamlit session generates a unique thread_id (uuid4).
• Sliding window ensures context window never exceeds LLM token limits.
• The full messages list is retained for session export / citation history.
• Multi-turn test: user asks 'Tell me about RL', then 'What did you say about reward shaping?' — agent must
correctly reference prior turn.

import streamlit as st
import uuid
from aria.graph import build_graph
st.set_page_config(page_title='ARIA — Research Intelligence', layout='wide')

    st.session_state.thread_id = str(uuid.uuid4())

            {'question': prompt, 'thread_id': st.session_state.thread_id,

            config=config
